## 1. 프로젝트 개요

### 1.1 프로젝트명
**DeReel** (Data Extraction & REEL Engine)  
> Apache Drill에서 영감을 받은 이름. 데이터를 깊이 파고들어(drill) 핵심 정보를 추출한다는 철학을 담음.

### 1.2 배경 및 목적
원하는 제품의 재고 여부 및 가격 변동을 수동으로 반복 확인하는 비효율을 해소하기 위해,  
여러 쇼핑 플랫폼을 자동으로 크롤링하여 조건 충족 시 즉시 알림을 발송하는 시스템을 구축한다.

본 프로젝트는 다음 두 가지 목적을 동시에 추구한다.
- **실용 목적:** 본인이 실제로 사용하는 개인용 자동화 알림 도구
- **교육 목적:** 크롤러 제작에 관심 있는 개발자를 위한 오픈소스 레퍼런스 및 블로그 연재 강좌 제공

### 1.3 프로젝트 범위
| 구분 | 내용 |
|---|---|
| **In Scope** | 재고 여부 감시, 가격 변동 감시, Telegram 알림, 가격 이력 저장, 웹 대시보드 (Phase 2) |
| **Out of Scope** | 자동 구매, 결제 연동, 로그인 필요 페이지 접근, 수집 데이터 제3자 재판매 |

## 2. 사용자 정의

### 2.1 주 사용자 (Primary User)
- **본인 전용 운영** — 초기에는 단일 사용자 환경으로 설계
- 감시 목록 직접 관리, 알림 채널 직접 설정

### 2.2 잠재 사용자 (Secondary User)
- GitHub 공개 저장소를 통해 코드를 접하는 개발자
- 블로그 연재 강좌를 따라 직접 크롤러를 구현해보고 싶은 입문자
- 향후 서비스화 시 일반 사용자 (멀티유저 구조는 별도 프로젝트로 분리 예정)

### 3.2 Phase 1 — Apple 리퍼비시 재고 감시

#### P0: 핵심 기능
- [ ] Apple 리퍼비시 스토어에서 **에어팟 제품군** 재고 여부 크롤링
- [ ] 신규 입고 감지 시 Telegram 알림 발송
- [ ] **1일 1회 알림 제한** — 동일 제품에 대해 24시간 내 중복 알림 없음
  - 입고 → 품절 → 재입고가 24시간 내에 반복되어도 최초 1회만 알림
- [ ] 크롤링 주기: **4시간 간격** (설정 파일로 변경 가능)
- [ ] 감시 제품 목록을 `targets.yaml` 파일로 관리

#### P1: 확장 기능
- [ ] Apple 리퍼비시 **전체 제품군**으로 감시 범위 확대
  - Mac, iPad, iPhone, Apple Watch, 액세서리 전체
- [ ] 품목 카테고리 필터 설정 지원

### 3.3 Phase 2 — 가격 변동 감시

지원 사이트 우선순위 (개발 순서):

| 순서 | 사이트 | 방식 | 이유 |
|---|---|---|---|
| 1 | **Steam** | 공식 API | API 제공, 구현 가장 단순 |
| 2 | **쿠팡** | 파트너스 API | 가입 완료, ToS 안전 |
| 3 | **GOG** | HTML 파싱 | 비공식 API 존재, 비교적 단순 |
| 4 | **Epic Games** | HTML 파싱 | 무료 게임 이벤트 알림 활용 가능 |
| 5 | **Amazon** | HTML 파싱 | 봇 탐지 강함, 마지막 구현 |

#### P0: 핵심 기능
- [ ] 제품별 **목표 가격 설정** — 현재 가격이 목표가 이하일 때만 알림
- [ ] **1일 1회 알림 제한** — 재고 감시와 동일한 규칙 적용
- [ ] 크롤링 주기: **3시간 간격** (설정 파일로 변경 가능)
- [ ] 사이트별 공식 권장 크롤링 주기가 있으면 **최우선 준수**

#### P1: 확장 기능
- [ ] 가격 **X% 이상 하락** 시 알림 조건 추가 지원 (목표가와 병행)
- [ ] Epic Games **무료 게임 배포 이벤트** 감지 및 알림

### 3.6 가격 이력 저장 및 집계

#### P0
- [ ] 크롤링 시점마다 가격 데이터 저장
- [ ] **보관 정책 (최대 5년)**:

| 데이터 나이 | 보관 방식 |
|---|---|
| 1개월 이내 | 수집 원본 그대로 보관 (raw) |
| 1개월 초과 ~ 1년 이내 | **월별 집계** (최솟값, 최댓값, 평균값) 로 압축 보관 |
| 1년 초과 ~ 5년 이내 | **연별 집계** (최솟값, 최댓값, 평균값) 로 압축 보관 |
| 5년 초과 | 자동 삭제 |

#### P1
- [ ] 집계 작업 자동화 스케줄러 구현 (월 1회 실행)
- [ ] 가격 이력 조회 API 제공 (대시보드 연계)

### 3.7 대시보드 (Phase 2)

#### P1
- [ ] 가격 트렌드 차트 (시계열)
- [ ] 재고 이력 타임라인
- [ ] 감시 목록 관리 UI

#### 대시보드 구현 방식 검토 (결정 필요)

| 방식 | 장점 | 단점 | 권장 단계 |
|---|---|---|---|
| **Grafana** | 빠른 구축, 차트 풍부, 무료 | 커스터마이징 제한 | Phase 2 초기 |
| **Kibana** | OpenSearch와 연동 강력 | 설정 복잡, 무거움 | Phase 3 (Kafka 도입 후) |
| **Vue.js 커스텀** | 완전한 커스터마이징 | 개발 공수 큼 | Phase 2 후기 또는 서비스화 시 |

> **권장:** Phase 2에서는 **Grafana**로 빠르게 대시보드를 구성하고,  
> Phase 3 (Kafka 파이프라인) 도입 후 Vue.js 커스텀 UI로 전환하는 2단계 전략 추천.

## 4. 비기능 요구사항

### 4.1 비용
- **Phase 1 목표: 월 $0** — GitHub Actions (공개 저장소) + Telegram Bot 무료 운영
- **Phase 2 목표: 월 $10 이하** — AWS Free Tier 범위 내 운영
- 유료 전환 시 사전 비용 검토 필수

### 4.2 안정성
- 크롤링 실패 시 다음 주기에 자동 재시도 (프로세스 중단 없음)
- 단일 사이트 오류가 전체 크롤러에 영향을 주지 않도록 격리

### 4.3 확장성
- 새로운 크롤러 추가 시 기존 코드 수정 최소화 (플러그인 구조)
- 향후 멀티유저, 서비스화 고려한 설계 (단, Phase 1에서는 구현 불필요)

### 4.4 법적/윤리적 준수
- 각 사이트의 `robots.txt` 및 이용약관 준수
- 서버 부하 방지를 위한 최소 요청 간격 준수
- 수집 데이터의 상업적 재판매 금지

### 4.5 보안
- API 키, Telegram 토큰 등 민감 정보는 GitHub Secrets 또는 `.env` 파일로 관리
- 민감 정보를 코드 또는 공개 저장소에 직접 커밋 금지

## 6. 개발 단계 로드맵

| Phase | 목표 | 핵심 기능 | 예상 기간 |
|---|---|---|---|
